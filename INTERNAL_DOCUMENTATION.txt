QUIZ PLATFORM - INTERNAL DOCUMENTATION
======================================

1. Project Summary
This project is a quiz platform with two main modes:
- Static quizzes for solo play without login
- Live quizzes hosted by admins with real-time room participation

The stack is split into:
- Frontend: React + React Router + Axios + Socket.IO client
- Backend: Node.js + Express + Socket.IO + MySQL
- Database: MySQL schema in database/schema.sql


2. Main User Flows

2.1 Student Static Quiz Flow
- User opens landing page
- Picks a preset quiz
- Enters name
- Answers all questions
- Frontend submits answers to backend
- Backend calculates score and stores attempt in StaticAttempts
- User is redirected to result screen

2.2 Student Live Quiz Flow
- User enters a 6-character room code on landing page
- User enters name and joins waiting room
- Host starts quiz
- Questions are pushed in real time over Socket.IO
- Student answers are submitted through sockets
- Backend stores each answer in Responses
- Final leaderboard is broadcast when quiz ends

2.3 Admin Flow
- Admin registers or logs in
- Can manage question bank
- Can create quizzes from selected questions
- Can create live rooms
- Can host live quiz sessions
- Can see room history and score summaries

2.4 Master Admin Flow
- Hardcoded internal credentials allow entry into a master dashboard
- Master admin can inspect all active live rooms
- Can end any room
- Can remove any live participant
- Can inspect static quiz attempts and delete them if needed

4. Important and Impressive Features

4.1 Dual Product Mode
The same platform supports:
- Solo static quizzes
- Multiplayer live hosted quizzes

This makes the project feel bigger than a single-use quiz app and gives it both demo value and practical flexibility.

4.2 Real-Time Live Quiz Hosting
The live quiz feature uses Socket.IO for:
- room creation
- participant join/rejoin
- question broadcast
- answer statistics
- pause/resume
- kick participant
- force end quiz
- final leaderboard broadcast

This is one of the strongest parts of the project because it makes the app feel interactive and event-driven instead of CRUD-based.

4.3 Host Recovery After Refresh
One important MVP decision implemented is host recovery:
- Active room metadata is stored locally in the host browser
- If the host refreshes, the UI reconnects and asks the server to resync room state
- The backend can rebuild the live room from the database if the room is not in memory

This is a strong product decision because live demos become much safer.

4.4 Database-Aware Live Room Rehydration
The backend does not rely only on RAM anymore for active rooms.
If needed, it can rehydrate a room using:
- Rooms
- Participants
- Responses
- Quizzes
- QuizQuestions
- QuestionBank
- Options

This improves resilience and makes the live quiz system feel more serious.

4.5 Master Admin Console
An internal master-admin flow was added for project-level control:
- inspect all active rooms
- remove live users
- end any room
- inspect static attempts
- delete static attempts

This is especially useful for demos, supervision, moderation, and emergency handling.

4.6 Clear Separation of Static and Live Data
A good architectural decision in this project is that solo play and live play use separate storage models:
- StaticAttempts for static quiz sessions
- Responses + Scores for live quiz sessions

This keeps business logic cleaner and avoids forcing very different workflows into one table design.

4.7 Route Protection for Admin Screens
Admin and master-admin routes now use a route guard, so protected screens do not silently open without a valid login context.

4.8 Schema Constraint Alignment
Another useful implementation detail:
- frontend quiz timer choices were aligned with backend/database constraints
- backend validates allowed timer values

This prevents silent DB errors during quiz creation.


5. Live Quiz Technical Notes
- Room state is primarily managed in backend/socket.js
- Active rooms are cached in liveRoomStore.js
- Progress is also persisted into the database for recovery
- current_question_index is stored in Rooms
- Answers are stored in Responses
- Final results are written to Scores
- Leaderboard broadcasts are computed from in-memory participant state backed by persisted answers


6. Security / Internal Notes
- Master admin credentials are intentionally hardcoded for internal operations
- This is okay for internal/demo use only
- For production, move them to environment variables and add proper audit logging


7. Known MVP Limitations
- Active live rooms still depend on the backend process being available
- FinalLeaderboard page route exists but is still placeholder-level
- Score controller routes are mounted but not fully implemented
- There is no full automated test suite yet
- Some internal admin pages are UI-heavy and rely on manual browser verification


8. Suggested Next High-Impact Improvements
- Real-time host analytics panel showing who answered, who is pending, and answer distribution
- Public post-quiz result/share page for finished live rooms
- Audit log for master-admin actions
- Full end-to-end test coverage for live room creation, join, refresh, pause, resume, and end


9. Recommended Startup Order
- Load database/schema.sql
- Load database/seed.sql
- Start backend server
- Start frontend app
- Use landing page for students
- Use /admin/login for admins and master admin
